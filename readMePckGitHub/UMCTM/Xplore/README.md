    <article class="markdown-body entry-content" itemprop="mainContentOfPage"><h1>
<a id="user-content-xplore" class="anchor" href="#xplore" aria-hidden="true"><span class="octicon octicon-link"></span></a>Xplore</h1>

<p>The aim of <code>Xplore</code> is to make generating publication quality, reproducible, and extensible plots and tables easy and fast. </p>

<h1>
<a id="user-content-contributing" class="anchor" href="#contributing" aria-hidden="true"><span class="octicon octicon-link"></span></a>Contributing</h1>

<p>The easiest way to contribute is to fork the repository, make changes and issue a pull request. How to do so is <a href="https://help.github.com/articles/fork-a-repo">documented here</a></p>

<h2>
<a id="user-content-relevant-files" class="anchor" href="#relevant-files" aria-hidden="true"><span class="octicon octicon-link"></span></a>Relevant Files</h2>

<ul class="task-list">
<li>
<code>README</code> - helpful information regarding contents and other information</li>
<li>
<code>DESCRIPTION</code> - brief description for R-package creation and release</li>
<li>
<code>NEWS</code> - tracks what changes are made for new releases</li>
</ul>

<h2>
<a id="user-content-workflow" class="anchor" href="#workflow" aria-hidden="true"><span class="octicon octicon-link"></span></a>Workflow</h2>

<p>Most contributions can be simply done by forking the repository and issuing a pull request with the updates, bug fixes etc.</p>

<p>Three scenarios that warrant creating a branch:</p>

<ul class="task-list">
<li><p>All long-term development should be conducted on the <code>dev</code> branch</p></li>
<li><p>While working on specific developments, please <a href="https://github.com/blog/1377-create-and-delete-branches">create a separate branch</a> with a name specific to the </p></li>
<li><p>While testing or making specific changes that may be hard to untangle use a <code>testing</code> branch</p></li>
</ul>

<h3>
<a id="user-content-versioning" class="anchor" href="#versioning" aria-hidden="true"><span class="octicon octicon-link"></span></a>Versioning</h3>

<p>Versioning follows a <a href="http://semver.org/">semantic versioning scheme</a></p>

<p>The cliff notes are as such:</p>

<p>Given a version number MAJOR.MINOR.PATCH, increment the:</p>

<p>MAJOR version when you make incompatible API changes,
MINOR version when you add functionality in a backwards-compatible manner, and
PATCH version when you make backwards-compatible bug fixes.</p>

<p>When a new version is appropriate, <code>git tag</code> to mark the version as well as bump <code>description</code> and <code>news</code> files</p>
